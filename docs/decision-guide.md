# 意思決定ガイド

設計が止まりやすい判断点と、止まったときの対処方針をまとめる。

---

## ADR-002（GC vs non-GC）が決まる前にできること

ADR-002 は最重要だが、プロトタイプが揃うまで測定できない。その間に進められる作業:

| 作業 | ADR-002 への依存度 |
|------|-----------------|
| パーサの実装 | なし |
| 型検査器の実装（型構成子・推論） | なし |
| enum / struct / pattern match の実装 | なし |
| `Option` / `Result` の実装 | なし |
| Wasm emit の基礎（i32/i64/f32/f64 演算） | なし |
| 名前解決・モジュールシステム | なし |
| `Vec[T]` の実装 | **強依存** → 後回し |
| クロージャの実装 | **強依存** → 後回し |
| `fs` API の実装 | 依存（wasi-resource-model も未決定） → 後回し |

ADR-002 を待たずに、パーサ → 名前解決 → 型検査 → Wasm emit（数値計算のみ）の縦一本を先に通すのが最速。

## ADR-002 の暫定決定ルール

`benchmark-plan.md` の比較測定が間に合わない場合は以下で暫定決定する:

> **LLM フレンドリを最優先に近く置くなら: Wasm GC を選ぶ**

根拠:
- Wado の実績（Hello World 1090 bytes、C 比 1.08–1.23x）から「GC でも目標を達成できる」ことが示されている
- ライフタイム管理を言語から取り除けるため、LLM の書き間違いが減る
- バイナリサイズも GC によりメモリ管理ルーチン不要で有利になりうる

**ただし暫定決定後にやること:**
- `language/memory-model.md` に「Wasm GC 前提」を記載
- `language/type-system.md` の ADR-002 待ちセクションを埋める
- `platform/wasm-features.md` の reference types / Wasm GC を Layer 1 に更新
- `stdlib/core.md` の String / Vec の実装方針を GC 前提で記載

## ADR-003（generics）の暫定決定ルール

ADR-002 の後に決める。ADR-002 で Wasm GC を選んだ場合:

> **制限付き monomorphization を選ぶ**（選択肢 A）

理由: Wasm GC 環境では uniform representation との相性が上がるが、数値性能を守るために値型は specialized にしたい。

制限:
- ネストしたジェネリクス（`Vec[Vec[T]]`）は v0 では禁止
- generic struct は `Vec` / `String` の専用実装のみ
- generic function は制限付き mono

## ADR-004（trait）の暫定決定ルール

> **v0 では trait なし**（選択肢 A）

条件: 以下を覚悟できるなら選択肢 A でよい
- `iter` と `HashMap` は v0 に入れない
- `for` 構文は v0 に入れない（または専用イテレータプロトコルを組み込みで持つ）
- 演算子オーバーロードは v0 に入れない

wado の実績では trait に相当する機能が v0 相当でもある程度存在している。「完全に trait なし」では B Tree Map の実装が不自然になる可能性があるため、以下の妥協案も検討:

> **v0 では「組み込みインターフェース」のみ。ユーザー定義 trait は v1 以降**

組み込みインターフェース候補:
- `Eq`（`==`/`!=` に必要）
- `Ord`（`<`/`>` に必要、B Tree Map の Key に必要）
- `Hash`（HashMap に必要、v1 以降でよい）
- `Display`（出力に必要）

## 「両対応」の誘惑を断る判断基準

以下のいずれかを感じたら「両対応」の誘惑に負けているサイン:

- 「GC があれば便利だが、non-GC も必要な場面がある」
- 「最初は両方試して後で決めたい」
- 「プロファイルで分ければ実装できる」

**対処**: ADR-002 に立ち返り、「暫定決定ルール」に従って一択に絞る。両対応は「二重実装」になる。コアが凍らない。

## 設計が止まったときのチェックリスト

```
□ ADR-002 は決定済みか?
  → No: benchmark-plan.md の条件を確認。揃わなければ暫定決定ルールを使う

□ ADR-002 の結果を language/, platform/, stdlib/ に反映したか?
  → No: 各ファイルの「ADR-002 待ち」セクションを更新する

□ ADR-003 は決定済みか?
  → ADR-002 の後に進める

□ v0 スコープに trait / iter / HashMap / async を追加しようとしていないか?
  → Yes: v0-scope.md の「入れないもの」を再読する

□ LLVM IR の都合で言語仕様を変えようとしていないか?
  → Yes: ADR-005 を再読する
```
