# ADR-014: Stability Labels for Language Spec and Stdlib API

**Status**: accepted  
**Date**: 2026-04-01  
**Deciders**: core team

## Context

Arukellt has a growing language spec and stdlib. Users need to know which features are safe to rely on in production code and which may change. Without explicit stability guarantees, users cannot make informed decisions about adoption risk.

## Decision

We adopt a four-level stability classification applied to all language features and all public stdlib API entries:

| Label | Definition |
|-------|-----------|
| `stable` | Specification frozen. Breaking changes only via migration guide with prior notice. |
| `provisional` | Broadly correct but minor details may change. No breaking changes without notice. |
| `experimental` | Active design work. Breaking changes possible without prior notice. |
| `unimplemented` | Specified but not yet implemented. Using it produces a compile error. |

### Application to language spec

- Every section in `docs/language/spec.md` carries a `<!-- stability: LABEL -->` comment after its heading.
- Section-level labels describe the majority of the section; individual items may have inline annotations.

### Application to stdlib

- Every public entry in `std/manifest.toml` has a `stability` field.
- Allowed values: `stable`, `provisional`, `experimental`.
- The stdlib reference docs generated by `scripts/generate-docs.py` show the stability label in the function table.

### Current baseline (2026-04-01)

**Language spec:**
- Sections 1–6, 8, 10 (lexical, types, expressions, statements, patterns, items, operator precedence): `stable`
- Section 7 (module system): `provisional`
- Section 9 (stdlib): see individual function entries

**Stdlib manifest:**
- All 267 public functions have explicit stability fields.
- Majority: `provisional`; no entries are `stable` yet (awaiting v1 freeze); none are `experimental` currently.

## Consequences

- Users can filter the stdlib reference by stability level.
- Compiler can emit warnings when `experimental` features are used (future work, tracked separately).
- When a feature is promoted from `provisional` to `stable`, a changelog entry is required.
- When a feature is demoted or removed, a migration guide entry is required.
