計算研究の信頼性は予期しない障害を管理できる実行設計に依存する。
research.do のような研究スクリプト運用と高負荷サーバー設計は障害管理という同じ課題を共有する。
再現可能な研究ではデータ処理と分析手順をコード化した実行フローが必要である。
Stata の .do ファイルは経済学や社会科学で標準的な分析実行単位として使われる。
巨大な単一スクリプトよりも 0-master.do と従属 .do ファイルに分割した構成が保守に強い。
0-master.do は実データ処理よりも環境変数定義と段階実行の制御を担うべきである。
1-merge.do と 2-stylized-facts.do と 3-regressions.do の分離は失敗時の再実行コストを下げる。
Stata 実行では log using logname.txt, replace text のようなテキストログ出力が監査性を高める。
プレーンテキストログは SCML よりも外部ツール連携と閲覧互換性が高い。
Do-file エディタの構文強調と折りたたみは forvalues や if/else の異常検出に有効である。
Stata では * コメントを使って入力データと出力物と理論前提を明示すると査読性が上がる。
研究パイプラインはローカル実行だけでなく GitHub Actions や GitLab CI に統合されるべきである。
マルチリポジトリ CI/CD は上流データ変更が下流分析に与える退行を自動検証できる。
LLM エージェントを使ったデータラングリング自動化は探索的分析の初期時間を短縮できる。
実行完了後のデータとコードは DOI 付き永続リポジトリへ固定アーカイブするべきである。
研究中のデータ保全には 3-2-1 バックアップルールの遵守が有効である。
障害対処の設計は予防中心のワークフロー設計と実行時反応の言語機構に分かれる。
Erlang は let it crash を採用し回復責任をスーパーバイザ層へ分離する。
Erlang のアクターモデルは共有メモリを避けメッセージパッシングで障害汚染を局所化する。
Erlang のスーパーバイザツリーはクラッシュしたプロセスを既知状態から再起動する。
let it crash は再試行可能なステートレス分散処理で特に効果を発揮する。
医療制御や重要インフラ制御ではクラッシュ許容より防御的設計が優先される。
Java は checked exception で回復可能エラー処理をコンパイル時に強制した。
Java の checked exception は API 進化時に throws 連鎖更新を誘発し互換性負担を増やす。
throws Exception や空 catch は checked exception 運用で生じやすい代表的アンチパターンである。
Java 8 の Streams とラムダは checked exception との合成で追加ラッパーを要しやすい。
C# は checked exception を採用せず開発速度と API 可変性を優先した。
C# 設計では例外の詳細型を中間層で列挙するより上位集中処理を重視する。
C++ 例外は stack unwinding と RAII に依存し安全性と性能の設計トレードオフが大きい。
C++ の基本保証は例外時でもリソースリークを防ぐことを要求する。
C++ の強い保証は失敗時の完全ロールバックを要求し copy-and-swap などのコストを伴う。
高性能領域では C++ 例外を無効化してエラーコードへ戻す運用が今も多い。
Python は LBYL より EAFP を推奨し例外捕捉で失敗経路を扱う。
EAFP は存在確認と実操作の時間差で起きる race condition を減らせる。
JavaScript の同期 try/catch は setTimeout など非同期コールバック内の例外を直接捕捉できない。
Promise の .catch() と async/await は JavaScript 非同期エラー処理の標準経路を提供する。
Go は errors are values を掲げて例外より多値返却の error を基本にする。
Go では if err != nil { return err } の明示分岐が障害経路を可視化する。
Go のエラー冗長性を減らす try/check 提案は隠れた制御フロー懸念で却下された。
Rust は Result<T, E> により回復可能エラーを型で表し未処理をコンパイルで拒否する。
Rust の match 網羅性検査は Err 分岐の取りこぼしを防ぐ。
Rust の ? 演算子は Err 時の早期 return と Into によるエラー変換を簡潔に実現する。
Rust の thiserror は公開ライブラリ向けの明示的エラー enum 設計に適する。
Rust の anyhow はアプリケーション向けに文脈付き集約エラー処理を簡略化する。
Rust の panic! は回復不能な契約違反に使うべきで回復可能エラーとは分離する。
Swift の throws は call site の try 明示で失敗可能地点の可読性を保つ。
Swift の実装は C++ 風の重い unwind ではなく戻り値ベース変換で効率を確保する。
Swift 6 の typed throws は throws(NetworkError) のような型付き伝播を可能にした。
Swift の fatalError() と preconditionFailure() は回復不能状態で即時停止を強制する。
Kotlin は checked exception を持たず runCatching による Result 連鎖を提供する。
Kotlin runCatching は CancellationException を握り潰すと structured concurrency を壊しうる。
Zig は error set と !T エラーユニオンで隠れた制御フローを排した設計を取る。
Zig の defer は常時クリーンアップを記述し errdefer はエラー時のみの巻き戻しを記述する。
Zig の unreachable は Debug では安全停止を促し ReleaseFast では誤用時 UB を招く。
Haskell の Maybe と Either は失敗をデータとして表現し Monad 合成で短絡伝播できる。
Result や Either の普及は例外の暗黙制御フロー問題を減らした。
Result ベース設計は呼び出し側へ効果が伝播する colored function 問題を生む。
代数的エフェクトは effect 宣言と handler 実装を分離して再開可能な失敗処理を提供する。
Koka と Frank と Unison は read や write や abort を型に載せる effect system を実装する。
Unison の Abilities は本番 I/O ハンドラとテスト用モックハンドラの差し替えを容易にする。
代数的エフェクト実装の主要課題は continuation 捕捉と再開に伴う性能コストである。
Koka は evidence-passing と選択的 CPS で重いランタイム依存を抑える。
Java Project Loom は Virtual Threads で thread-per-request の高コスト問題を緩和する。
Virtual Threads はブロッキング時に JVM が unmount しキャリアスレッドを再利用できる。
I/O 完了後の remount により Java は同期記述のまま高並行性を得られる。
Project Loom の Structured Concurrency は親子タスクのライフサイクルを統合管理する。
親タスク失敗時に従属仮想スレッドを一括終了できるためリークと孤児タスクを減らせる。
堅牢な計算システムにはモジュール化実行と明示的エラー表現と回復戦略の分離が必要である。
